# Candidate Client Handoff

将一段候选人聊天记录和一个候选人文件整理成可直接转发给客户的中文提报摘要，并交付标准命名的 PDF 简历附件。

## 工作流

1. 完整读取用户提供的聊天记录和候选人文件。聊天记录可以是粘贴文本、截图、导出文件或转录；候选人文件可以是 PDF、Word、图片或其他简历/个人资料格式。
2. 识别候选人姓名。优先采用候选人本人资料页或简历身份区中的姓名；用户明确纠正时采用用户最新口径。不要把招聘方、推荐人、公司、学校、邮箱前缀或文件标签误认为候选人姓名。
3. 按“证据与冲突规则”提取全部核心字段。核心字段不得省略；信息缺失时填写 `未提及`，存在冲突或歧义时填写 `待确认` 并简述冲突。
4. 判断候选人是否具有大厂经历，并按“大厂职级和绩效”规则决定是否增加条件字段。
5. 仅基于真实客户/岗位上下文生成匹配建议：
   - 优先使用用户本轮提供的客户、JD、岗位要求或明确指定的本地岗位上下文。
   - 识别不同公司，并按“多公司消息拆分”规则确定需要生成的消息。
   - 用户明确表示多家公司都可投递、都推进或全部处理时，为每一家被确认的公司生成消息，不限制为 3 家。
   - 用户仅要求推荐、尚未确认全部推进时，按匹配度选择最多 3 家公司。
   - 同一家公司存在多个岗位时，默认选择最匹配的岗位；用户明确要求多个岗位时，在该公司的同一条消息中列出。
   - 只有候选人信息、没有真实客户上下文时，客户填写 `待匹配`，岗位可填写有证据支持的方向；不得编造客户名称、在招岗位或招聘需求。
   - 需要正式打分或在多个真实岗位间比较时，使用 `$candidate-fit-tracker` 的角色上下文和证据规则。
6. 将候选人文件转换并整理为 PDF：
   - 姓名可靠确认后，使用 `$ttc-pdf-normalizer` 生成 `<候选人姓名>-TTC.pdf`。
   - 保留原文件，不移动、不删除、不改写简历内容。
   - 最终文件名只包含候选人姓名、一个 ASCII 连字符、`TTC` 和 `.pdf`，例如 `王小明-TTC.pdf`。
   - 文件名已带 `-TTC` 时不要重复追加。
   - 若姓名不可靠或多个姓名均可能，先输出可完成的提报草稿，再询问姓名；确认前不得生成最终 PDF。
   - 若同名目标文件已存在且内容不同，不得覆盖，除非用户明确授权。
7. 打开并检查最终 PDF。确认可读取、至少一页、页面非空，且没有明显缺字、裁切、乱码、方向错误或分页破坏。
8. 使用固定模板交付。除非用户要求分析过程，否则不要添加评分、泛化评价、面试建议或额外章节。

## 证据与冲突规则

- 只写来源中明确存在的信息。不得猜测离职原因、薪酬、期望、所在地、意向地、到岗时间、竞业限制或其他面试流程。
- 对当前状态、求职动机、薪酬、地点意向、到岗时间和其他流程，按以下优先级取值：用户最新明确说明 > 最新聊天记录 > 较早聊天记录 > 简历。
- 对姓名、教育、任职公司、岗位、项目和日期，优先采用候选人文件中的身份区和经历区；聊天记录或用户明确纠正时更新。
- 若聊天记录和文件冲突且无法判断新旧，写 `待确认：聊天记录为…；简历为…`，不要静默选择一方。
- 区分事实和推断。岗位匹配理由可以是基于事实的判断，但不得把判断写成候选人经历事实。
- 保留原始单位、币种、税前/税后口径和时间口径。分别写当前薪酬与期望薪酬；没有明确依据时不要自行换算总包。
- 仅在组成项与计算关系明确时做简单合计，并注明 `按已提供组成估算`。股票、期权、奖金和补贴不得当作确定现金收入。
- `其他在面流程或限制` 可包含其他公司面试阶段、offer/截止时间、竞业、通知期、签证/工签、合同约束、远程/出差/加班限制等；未提及统一写 `未提及`。
- 涉及敏感个人信息时只保留本次提报所需内容，不输出身份证号、完整住址、私人联系方式等无关信息。

## 大厂职级和绩效

- 当候选人的当前或过往雇主属于公认的大型互联网、科技、通信、消费电子或平台型企业，或用户明确称其为大厂经历时，增加 `职级和绩效` 字段。
- 无大厂经历时完全省略该字段，不要输出 `不适用`、`无` 或占位标题。
- 大厂候选人即使没有提供职级或绩效，也必须保留该字段，分别填写 `职级：待确认`、`绩效：待确认`。
- 优先写最近一段大厂经历的职级和最近绩效；若信息来自更早的公司或周期，注明公司和年份/考核周期。
- 保留公司内部原始职级写法，例如 `P7`、`T3-2`、`L5`、`17级`。绩效同时保留评级与周期，例如 `2025年度 A`。
- 不得根据职位名称、工作年限、薪酬、汇报关系、晋升经历或团队规模推断内部职级。
- 不得根据奖金、调薪、晋升或项目成果推断绩效评级。

## 多公司消息拆分

- 只有一家公司时，输出一条完整提报消息。
- 存在多家目标公司时，每家公司输出一条独立、完整、可直接复制转发的提报消息。
- 每条消息的 `建议客户、岗位` 只能出现当前公司，不得把其他公司或岗位混入同一条消息。
- 根据当前公司的 JD 单独改写匹配依据和风险；姓名、状态、薪酬、职级绩效、地点、到岗时间、其他流程和 PDF 链接可以在各消息中重复。
- 按匹配度从高到低排列公司消息；用户指定顺序时遵循用户顺序。
- 使用 `### 消息 N｜<公司名>` 标识每条消息，并用水平分隔线 `---` 分隔。标签不属于提报正文。
- 不要先输出一个合并版摘要，也不要在所有消息之后再次汇总公司名单。

## 固定输出模板

每一条公司消息始终严格保留下列 8 个核心粗体字段标题及顺序：

```markdown
**候选人姓名**
<姓名；不可靠时写“待确认”>

**建议客户、岗位**
<仅写当前公司｜岗位；简短匹配依据。没有真实客户上下文时写“待匹配｜岗位方向”>

**看机会原因 / 当前状态（离职/在职）**
<在职/离职/待确认；看机会原因。两项分别说明>

**当前薪酬和期望薪酬**
<当前：…；期望：…>

**所在地和意向地**
<所在地：…；意向地：…>

**到岗时间**
<明确日期、周期或“未提及”>

**其他在面流程或限制**
<其他面试/offer/通知期/竞业/工签/工作方式限制；无信息写“未提及”>

**简历附件PDF（命名：姓名-TTC）**
[<姓名>-TTC.pdf](<最终文件的绝对路径>)
```

候选人有大厂经历时，在 `当前薪酬和期望薪酬` 与 `所在地和意向地` 之间插入以下字段；无大厂经历时不插入：

```markdown
**职级和绩效**
<职级：…；绩效：…。缺失项写“待确认”>
```

## 交付要求

- 默认使用简洁、客观、可直接转发的中文。
- 每个字段控制在必要长度内。
- 多家公司必须按公司拆成多条完整消息，不得合并在一条消息中。
- 用户明确同意全部推进时，覆盖所有已确认公司；用户只要求推荐时，最多输出 3 家。
- 大厂候选人必须包含 `职级和绩效`；非大厂候选人不得包含该字段。
- 同一份最终 PDF 链接可以在每条公司消息中重复。
- 最终 PDF 已生成时必须提供可点击的本地绝对路径链接。
- PDF 尚未生成时，在附件字段写清唯一阻塞项，例如 `待确认候选人姓名后生成`，并只询问完成文件所必需的问题。
- 不要声称 PDF 已生成、已转换或已验证，除非对应操作实际完成。
